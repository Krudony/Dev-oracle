## What Could Improve
* The CLI should ideally handle blank or invalid JSON files gracefully by fallback or warning, rather than crashing the entire discovery process.
* Keep standard instructions for WSL2 networking handy since virtual switch host IPs tend to shift or block ports by default.

## Honest Feedback (100+ words, 3 friction points)
1. **Config Fragility:** The JSON parser in `antigravity-cli` (particularly `discovery.go`) is too strict. A 0-byte file should be treated as an empty config rather than breaking the loop. It took scanning raw CLI logs to figure this out, which might be obscure for general users.
2. **Path Scattering:** Having configuration values split between `config.toml` (Hermes), `settings.json` (Antigravity), and an empty `mcp_config.json` creates unnecessary confusion and scatter.
3. **WSL2 Network Barriers:** Connecting back to the Windows host for services like Ollama via raw virtual ethernet IPs (`172.30.128.1`) remains highly unstable and subject to firewall changes and binding setups.

## Lessons Learned
1. **0-Byte Files Break Parsers:** Never trust the CLI to bypass empty config files. Keep at least `{}` inside all JSON configs to ensure parsing doesn't crash the pipeline.
2. **Logs Over Speculation:** Checking file existence is not enough; raw execution dry-runs (`bun run`) and API timeouts (`curl`) provide absolute truths.
